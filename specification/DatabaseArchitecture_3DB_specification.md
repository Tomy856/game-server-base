# 3DB アーキテクチャ仕様書

## 1. 概要
- **目的**: ゲームサーバーのスケーラビリティと運用性を考慮した3DB構成（センターDB、マスターデータDB、ユーザーデータDB）の設計。ゲストログイン、引継ぎ、シャーディング、バージョン管理をサポート。
- **構成**: 
  - センターDB: 課金・セッション・引継ぎ情報管理（単一インスタンス）
  - マスターデータDB: ゲーム定義データ（読み取り専用、バージョンDB: master_data_v1, v2...）
  - ユーザーDB: プレイヤーデータ管理（シャーディング前提、user_db_01, user_db_02からスタート、負荷に応じて拡張）
- **技術スタック**: PostgreSQL, Redis (キャッシュ), Go (Gin framework)
- **影響範囲**: 全API（ゲスト作成、引継ぎ、セッション管理）、DBマイグレーション、キャッシュ戦略

## 2. 設計詳細
- **アーキテクチャ図**:
  ```
  [クライアント] → [ゲームサーバー (Go/Gin)]
    ↓
  [センターDB (課金・セッション)] ← 単一
  [マスターデータDB (v1, v2...)] ← バージョン管理
  [ユーザーDB (01, 02...)] ← シャーディング (UUID % N)
    ↓
  [Redis (キャッシュ)] ← セッションTTL=1h, マスターデータアクセス時キャッシュ
  ```
- **相互依存関係**: センターDBのUUID生成がユーザーDB操作の前提。マスターデータは読み取り専用でキャッシュ依存。

## 3. 実装関連
- **マイグレーション**: golang-migrate使用、フォルダ分離 (db/migrations/center/, user/, master/)
- **既存機能への影響**: なし（新規DB構成）
- **DB変更**: 新規テーブル作成、シャーディング準備

## 4. 並行制御・ロック機構
- **ロック戦略**: ユーザーDBにu_lockテーブルを用意、uuidでSELECT ... FOR UPDATEで排他ロック
- **タイムアウト**: ロック取得タイムアウト=5秒
- **デッドロック防止**: ロック順序を固定（uuid昇順）

## 5. トランザクション管理
- **複数リソース更新順序**: センターDB先行、ユーザーDB後続
- **一貫性確保方法**: センターDB UUID生成成功基準、失敗時は管理画面で補正（INSERT/UPDATE）
- **ロールバック戦略**: 部分失敗時は手動リトライ

## 6. エラーハンドリング
- **起こり得るエラー**: DB接続失敗、ロックタイムアウト、UUID重複
- **対応方法**: エラーログ記録、管理画面通知、クライアントにエラーメッセージ返却

## 7. パフォーマンス・スケーラビリティ
- **大規模アクセス時の対応**: ユーザーDBシャーディング拡張、マスターデータ全キャッシュ
- **レスポンスタイム目標**: APIレスポンス<200ms
- **スループット要件**: 同時接続1000件対応

## 8. ロギング・監視項目
- **ログイベント**: UUID生成、引継ぎ成功/失敗、ロック競合
- **監視**: DB接続数、Redisヒット率、ロック待機時間

## 9. テストケース

### ケース 1: ゲスト作成正常系
- **入力**: 新規ゲストリクエスト
- **期待結果**: センターDBにUUID生成、ユーザーDBに初期値INSERT、クライアントにUUID返却
- **テスト方法**: 統合テストでDB確認、RedisセッションTTL検証

### ケース 2: 引継ぎ正常系
- **入力**: ゲストUUID + Google認証（provider_idハッシュ）
- **期待結果**: transfer_credentialsに保存、クライアントにUUID返却、ゲストデータ保持
- **テスト方法**: ユニットテストでハッシュ検証、DB更新確認

### ケース 3: ロック競合異常系
- **入力**: 同時ゲストログイン（同一UUID生成試行）
- **期待結果**: 一方がロック待機、他成功、エラーログなし
- **テスト方法**: 並行テストでタイムアウト確認、DB重複なし

### ケース 4: セッションタイムアウト異常系
- **入力**: 1時間未アクセス後APIコール
- **期待結果**: ログアウト扱い、エラーレスポンス
- **テスト方法**: Redis TTLテスト、APIレスポンス検証

### ケース 5: マスターデータバージョン更新正常系
- **入力**: 新バージョンDB作成、ネイティブアプリにバージョン通知
- **期待結果**: アプリ再ダウンロード、旧バージョンDB保持
- **テスト方法**: DBクエリテスト、バージョン通知API確認
