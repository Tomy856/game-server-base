# Game Server Base

このリポジトリは、**ドメイン駆動設計（DDD）** と **クリーンアーキテクチャ** に基づいて構築されたゲームサーバープロジェクトの雛形です。

## 構造と設計方針

### 🎯 基本理念
- **ドメイン中心**: ビジネスロジックを中心に据え、他の層から独立させます。
- **層の分離**: 各層は明確な責務を持ち、依存関係は内側から外側へ向かうようにします。
- **技術非依存**: ドメイン層は具体的なフレームワークやインフラに依存せず、テストや移植性が高い設計を目指します。

### 📁 フォルダ構成
プロジェクトルート以下に主要なディレクトリを配置しています。各ディレクトリの役割は以下の通りです。

```
internal/
├── domain/              # ドメイン層（ビジネスルール）
│   ├── entity/          # エンティティ
│   ├── value/           # 値オブジェクト
│   ├── repository/      # リポジトリのインターフェース（契約）
│   └── service/         # ドメインサービス
├── application/         # アプリケーション層（ユースケース）
│   ├── usecase/         # ユースケース実装
│   ├── service/         # アプリケーションサービス
│   └── dto/             # データ転送オブジェクト
├── infrastructure/      # インフラ層（実装詳細）
│   ├── persistence/     # DBリポジトリなど
│   ├── cache/           # キャッシュ実装
│   └── external/        # 外部APIクライアント等
└── presentation/        # プレゼンテーション層（APIなどの入出力）
    ├── handler/         # HTTPハンドラー（コントローラー）
    ├── router/          # ルーティング設定
    └── middleware/      # ミドルウェア

pkg/                     # 外部共有パッケージやユーティリティ
```

#### フォルダの意図まとめ
- **domain**: 変更されにくいビジネスルールを収める。
- **application**: ユースケースやDTOなど、ドメインに対する操作を定義。
- **infrastructure**: 具体的な技術（DB、キャッシュ、外部サービス）への接続。
- **presentation**: APIエンドポイントやレスポンス形式を担う。
- **pkg**: チーム全体で使う汎用ライブラリ。

### 🛠 技術スタック
- 言語: Go（Ginフレームワーク想定）
- DB: PostgreSQL
- キャッシュ: Redis / Memcached

### 📦 プロジェクト開始手順
1. リポジトリをクローン
2. 依存関係を `go mod tidy` で整理
3. `docker-compose` で DB/Redis を起動
4. `go run main.go` でサーバーを立ち上げ

詳細なセットアップや開発ルールは随時このREADMEに追記します。
